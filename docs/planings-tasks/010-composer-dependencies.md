# Planning: Composer Dependencies — PHP 8.1 + BerlinDB + Access Control + Main Menu (Feature 010)

## Context

Post-migration audit (2026-07-01) surfaced three composer-side issues that need attention before the `feature/issue-3 → main` cutover:

1. **PHP version mismatch** — `composer.json` declares `"php": ">=7.4"` but the plugin header + `README.txt` say `Requires PHP: 8.0`. A developer installing on PHP 7.4 gets a successful `composer install` followed by an activation failure. This feature corrects the drift AND bumps the baseline to **PHP 8.1** (all sync points).
2. **Optional-but-required packages have no composer entry** — `\WPBoilerplate\AccessControl\AccessControlManager` is consumed in 4 files (guarded by `class_exists()`) but not declared as a require. The plugin technically "works without it" but the admin surface degrades. Same story for `berlindb/core` (referenced by name only, not imported — custom Query classes mimic its shape).
3. **`acrossai-co/main-menu` migration deferred** — the plugin's admin menu (`admin/Partials/Menu.php`) still uses raw WordPress `add_menu_page` / `add_submenu_page` calls. The `acrossai-co/main-menu` package (v0.0.8) is the standard menu-builder across other AcrossAI plugins; this feature migrates our admin menu onto it.

**Outcome after this feature**:
- `composer.json` reflects the actual runtime dependency graph
- PHP 8.1 baseline is uniformly declared (composer.json + plugin header + README.txt + constitution + copilot-instructions)
- Admin menu registered via `acrossai-co/main-menu`; the plugin's admin URL `?page=acrossai_mcp_manager` is preserved for external bookmarks
- `wpb-access-control` becomes a hard require (but `class_exists()` guards STAY in place as defense-in-depth per CONSTRAINTS below)
- `berlindb/core` is available in `vendor/` but the custom Query classes remain — a real-BerlinDB refactor is scoped as a separate future feature

## CONSTRAINTS

**Do not** — the following are absolute for Feature 010; violation = revert:

1. **Do not remove `class_exists( '\WPBoilerplate\AccessControl\AccessControlManager' )` guards in this feature.** Even though the package becomes a hard require, the guards stay as defense-in-depth against runtime upgrade issues (e.g. `composer install` runs on a machine where the package fails to autoload for some reason). Removal is scoped to a separate feature after 3+ months of soak time.

2. **Do not refactor `Includes\Database\...\Query` classes to consume real BerlinDB in this feature.** Custom classes stay. Real-BerlinDB migration is a semantic change requiring its own test coverage sweep — scoped as Feature 011 per TASK-8.

3. **Do not introduce PHP 8.1-specific language features** (`readonly` properties, `enum`, `never` return type, first-class callable syntax) in files touched this feature. The bump raises the minimum runtime; language-level modernization is a separate concern.

4. **The PHP version bump MUST be atomic.** composer.json + plugin header + README.txt + constitution + copilot-instructions all change in the SAME commit. Splitting produces a window where a partial upgrade breaks activation.

5. **The admin menu URL `?page=acrossai_mcp_manager` MUST be preserved.** External bookmarks, Application Password callbacks, integration tests, and documentation all depend on this URL. If `acrossai-co/main-menu` forces a different URL structure, escalate — do not silently change the slug.
