## Notes for `/speckit-clarify`

The 5 clarification-worthy items surfaced by the spec's `Assumptions` block:

- **SAFE_CAPABILITIES enumeration** — the initial list is FR-009 (edit_posts, publish_posts,
  read, moderate_comments, manage_categories, upload_files). Confirm this list is complete
  for the initial ship OR add operator-selectable additions before implementation.
- **User autocomplete implementation** — FR-009 specifies `wp-user-search` REST endpoint.
  Confirm this returns display_name + id (needed for pill rendering) OR fall back to
  `wp_dropdown_users()` (simpler but less friendly for large user bases).
- **`gate_mcp_tool_call` behavior on `MCPServerQuery::get_item()` null** — spec's edge case
  says fail-open (return `$args`). Confirm this is the right default vs. fail-closed
  (return WP_Error). Fail-open is the current recommendation because a stale server_id
  during a concurrent DELETE race is more likely than a malicious spoof.
- **Version option cleanup on opt-in** — FR-012 deletes `wpb_ac_mcp_manager_db_version`.
  Confirm the vendor package doesn't rely on this option existing for any post-uninstall
  hook (unlikely, but worth verifying by reading `RuleTable::maybe_upgrade()`).
- **Third-party provider filter timing** — FR-014 wires `add_filter( PROVIDERS_FILTER, ...
  'register_default_providers' )` in Main.php. Confirm the filter fires late enough that
  third-party plugins hooked at their own `plugins_loaded` can also append. The v2
  package fires this filter on `init` priority 5, so any third-party plugin loaded
  before `init` can hook it.
