# Feature Specification: OAuth Per-Server Scoping

**Feature Branch**: `032-oauth-per-server-scoping`
**Created**: 2026-07-21
**Status**: Draft
**Input**: User description: The AI Connectors feature (F021 + F024) has a cross-server scoping gap. An admin editing Server A's Connectors tab can revoke or delete Server B's OAuth clients and tokens by modifying `client_id` in the outbound REST body — nothing at the SQL or REST layer prevents it. Additionally the same DCR connector (e.g. Claude Desktop) cannot be registered on multiple servers because `UNIQUE(client_id)` is global. Fix by adding `server_id` as a first-class column on the three OAuth tables (via BerlinDB D28 3-part contract), backfilling from the `server-{id}-` naming convention, changing to composite `UNIQUE(client_id, server_id)`, propagating `server_id` through the OAuth flow, and requiring + validating `server_id` on every mutating REST endpoint. See `docs/planings-tasks/032-oauth-per-server-scoping.md` for the full engineering brief.

## Clarifications

### Session 2026-07-21

- Q: Observability signal for cross-server bypass attempts (403 `acrossai_mcp_oauth_cross_server` responses)? → A: Fire `do_action( 'acrossai_mcp_oauth_cross_server_attempted', $client_id, $server_id_requested, $server_id_actual, get_current_user_id(), time() )` on every 403 response — D19 fail-open observability precedent matching F015 / F020 / F029 / F030.
- Q: Rollout — ship the fix unconditionally, or behind a feature flag with an opt-out window? → A: Ship unconditionally. The cross-server bypass is a security vulnerability, not a documented feature; a legacy-mode toggle would legitimise the vulnerable behaviour and permanently branch the code. Rollback is via composer downgrade if operationally required. Pre-flight attestation (SEC-032-001) is captured at plan phase as awareness, not a fallback trigger.
- Q: Legacy pre-F032 DCR rows (no `server-{id}-` prefix, `server_id = NULL` after migration) — leave invisible-but-alive, or auto-delete? → A: Auto-delete unconditionally during the upgrade callback (A-aggressive form). The OAuthClients upgrade callback purges every row with `server_id = NULL` after the prefix-backfill step, and the OAuthTokens + OAuthAuthCodes upgrade callbacks purge every row whose `server_id` remains `NULL` after their JOIN backfill. This silently disconnects any live AI-host session bound to a pre-F032 DCR row on the next request; users on Claude.ai / ChatGPT / etc. re-run the OAuth authorize flow to reconnect. The tradeoff is accepted intentionally: leaves the table fully server-scoped post-migration with zero orphan rows, no follow-up WP-CLI needed, no permanent "legacy" panel in the UI. README + release-note MUST prominently warn operators before upgrade. Purge counts fire `do_action( 'acrossai_mcp_oauth_legacy_dcr_purged', int $clients_deleted, int $tokens_deleted, int $auth_codes_deleted )` once per upgrade run for operator observability. This reverses the planning doc's original "do not delete legacy DCR rows" CONSTRAINT — the planning doc MUST be updated at `/speckit-plan` phase to match.
- Q: `server_id` column nullability post-migration — `NOT NULL` from day one, or stay `NULL`-allowed? → A: `NOT NULL` from day one on all three OAuth tables. The auto-purge from Q3 guarantees zero `server_id IS NULL` rows post-migration, so the schema itself enforces the invariant at the SQL layer (stronger than PHP-only enforcement, matches how `client_id` is declared, closes the class of defensive-check-forgotten bugs). Upgrade callback ordering per table becomes: (1) ADD COLUMN `server_id BIGINT UNSIGNED NULL` if missing, (2) backfill, (3) purge remaining NULLs, (4) `ALTER TABLE ... MODIFY server_id BIGINT UNSIGNED NOT NULL` (guarded by `INFORMATION_SCHEMA.COLUMNS.IS_NULLABLE = 'YES'` check for idempotency). OAuthClients also adds the composite UNIQUE step between (3) and (4). Schema.php declares the final NOT NULL state so BerlinDB's SHOW CREATE TABLE match check on subsequent boots stays green.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Cross-server privilege escalation is impossible (Priority: P1 — security)

A site administrator manages two MCP servers, Server A and Server B, each with its own set of OAuth-authorized AI clients (Claude.ai, ChatGPT, etc.). The admin opens Server A's Connectors tab and clicks "Revoke tokens" or "Delete" on one of Server A's clients. The action affects ONLY Server A's client/tokens — Server B's clients and tokens for any connector remain untouched, no matter what `client_id` is submitted in the REST request. If a modified request body attempts to name a `client_id` bound to Server B, the REST endpoint returns 403 with a specific error code (`acrossai_mcp_oauth_cross_server`) without leaking cross-server existence information.

**Why this priority**: this is the security fix the feature exists to deliver. Without it, an operator on any single server can compromise every other server on the site via the OAuth admin surface.

**Independent Test**: Seed two MCP servers each with a Claude.ai client and issued tokens. From Server A's Connectors tab, POST `/wp-json/acrossai-mcp-manager/v1/oauth/revoke-client-tokens` with `client_id` set to Server B's client + `server_id` set to Server A → assert 403 `acrossai_mcp_oauth_cross_server`. From same tab, POST the correct `server_id + client_id` pair for Server A → assert 200 + only Server A's tokens revoked in DB.

**Acceptance Scenarios**:

1. **Given** two MCP servers each with an admin-generated Claude.ai OAuth client + issued tokens, **When** an admin on Server A's Connectors tab clicks Revoke on their Claude client, **Then** ONLY Server A's tokens are revoked (Server B's tokens for its distinct Claude client row remain `revoked = 0`) AND the response is 200 + `revoked_count` reflecting Server A's count only.
2. **Given** the same seed, **When** an authenticated `manage_options` user POSTs `revoke-client-tokens` with `server_id = 1` (Server A) but `client_id` pointing at Server B's client, **Then** the response is 403 with error code `acrossai_mcp_oauth_cross_server` AND Server B's tokens remain active AND the response body does NOT confirm existence of the cross-server client.
3. **Given** the same seed, **When** the same modified request targets `delete-client` instead of revoke, **Then** same 403 outcome AND Server B's client row remains in `oauth_clients`.
4. **Given** a request missing `server_id`, **When** any mutating OAuth REST endpoint receives it, **Then** the response is 400 `invalid_request` (not 403) — the missing-param path is distinct from the cross-server path.

### User Story 2 — Same DCR connector registers on multiple servers as distinct rows (Priority: P1 — core functionality)

A site has two MCP servers. An AI host that supports Dynamic Client Registration (Claude Desktop) is installed on the operator's machine and configured to authorize against Server A. It sends a DCR request to the plugin's registration endpoint with an RFC 8707 `resource` URL pointing at Server A. The registration succeeds and creates a new client row bound to Server A. The operator later configures the same Claude Desktop instance to also authorize against Server B (same client, different resource URL). The second DCR request succeeds and creates a **separate** client row bound to Server B — Claude Desktop is now registered independently on both servers, with independent tokens, revoke controls, and delete controls per server. (Any pre-F032 legacy DCR row for Claude Desktop has already been purged by the upgrade — see User Story 3 scenario 3.)

**Why this priority**: without this, the plugin CANNOT correctly serve multiple MCP servers on the same site when DCR-based AI hosts are in use — the second registration either fails (composite conflict on `client_id`) or silently reuses the first server's client, both of which are broken.

**Independent Test**: On a fresh install with two seeded MCP servers, POST two DCR registrations to the plugin's DCR endpoint with `resource` URLs targeting each server; assert two distinct rows exist in `oauth_clients` with the same `client_name` (Claude Desktop) but distinct `server_id` values.

**Acceptance Scenarios**:

1. **Given** a fresh install with two MCP servers (server_id = 1, server_id = 2), **When** two DCR requests arrive with the same `client_name = "Claude Desktop"` but `resource` URLs targeting each server, **Then** two distinct rows exist in `oauth_clients` — both with `client_name = "Claude Desktop"`, distinct `client_id` values, distinct `server_id` values (1 and 2 respectively) — AND the composite `UNIQUE(client_id, server_id)` accepts both without conflict.
2. **Given** two DCR clients as above with issued tokens each, **When** an admin revokes tokens for Server A's Claude Desktop, **Then** Server B's Claude Desktop tokens remain active.
3. **Given** a DCR request with a `resource` URL that does not resolve to any registered MCP server, **When** the DCR endpoint processes it, **Then** the response is 400 `invalid_target` — the client row is NOT created (fail-closed).
4. **Given** a DCR request with an empty `resource` param, **When** the endpoint processes it, **Then** the response is 400 `invalid_target` (RFC 8707 mandates resource binding on the plugin's OAuth surface).

### User Story 3 — Legacy OAuth data migrates safely on upgrade (Priority: P2 — operator confidence)

An operator upgrades the plugin from a version predating F032 to the F032-shipped version on a live install that has pre-existing OAuth clients, tokens, and auth codes. On the next admin page load after deploy, the three OAuth tables gain the `server_id` column via BerlinDB `$upgrades` callbacks — existing admin-generated clients are backfilled from the `server-{id}-{slug}-{rand}` prefix, existing tokens + auth codes are backfilled via JOIN on `client_id`, and any remaining rows whose `server_id` is still `NULL` after backfill (legacy pre-F032 DCR clients + their tokens + their auth codes) are PURGED from the tables. The composite `UNIQUE(client_id, server_id)` index replaces the standalone `UNIQUE(client_id)` in the correct order (ADD composite → DROP old) so the table is never unconstrained mid-migration. Live AI-host sessions bound to legacy DCR rows are disconnected on the next request; users re-run the OAuth authorize flow to reconnect (README + release-note warn operators of this consequence prominently). No admin action is required post-upgrade. `debug.log` shows no warnings. Purge counts fire `do_action( 'acrossai_mcp_oauth_legacy_dcr_purged', $clients_deleted, $tokens_deleted, $auth_codes_deleted )` for operator observability.

**Why this priority**: safe in-place upgrade is the D28 mandatory contract. Skipping this scenario would recreate the F029 `B34` silent-write-loss failure mode (schema stamped without ALTER applied). The auto-purge is accepted as a deliberate one-time cost for a fully server-scoped table post-migration (see clarification 2026-07-21 Q3).

**Independent Test**: Take a snapshot of the OAuth tables from a pre-F032 install (populated with admin clients + DCR clients + tokens + auth codes), deploy F032, load any wp-admin page, verify (a) the three `db_version_key` options advance, (b) `DESCRIBE` on each table shows the `server_id` column, (c) admin-client rows have `server_id` backfilled from the prefix, (d) legacy DCR client rows are DELETED (SELECT COUNT WHERE server_id IS NULL = 0), (e) tokens + auth codes bound to purged DCR clients are also deleted (SELECT COUNT WHERE server_id IS NULL = 0), (f) tokens + auth codes bound to admin clients have `server_id` backfilled via JOIN, (g) composite `UNIQUE(client_id, server_id)` is present on `oauth_clients` and the standalone `UNIQUE(client_id)` is gone, (h) `acrossai_mcp_oauth_legacy_dcr_purged` action fired exactly once with correct deleted counts.

**Acceptance Scenarios**:

1. **Given** a pre-F032 install with admin OAuth clients (`client_id` matching `/^server-\d+-/`), **When** the operator loads any wp-admin page after deploying F032, **Then** the `wpdb_acrossai_mcp_oauth_clients_version` option advances AND every admin client row has `server_id` populated from its prefix AND no `debug.log` errors are emitted.
2. **Given** the same install with issued OAuth tokens + auth codes for those admin clients, **When** the upgrade fires, **Then** every token + auth_code row whose client has a resolved `server_id` also has `server_id` populated via JOIN backfill.
3. **Given** pre-F032 DCR client rows (no `server-{id}-` prefix) + their tokens + their auth codes, **When** the upgrade fires, **Then** those DCR client rows are DELETED after the backfill step AND their tokens are DELETED AND their auth codes are DELETED AND `acrossai_mcp_oauth_legacy_dcr_purged` fires exactly once with the deleted counts. Live AI-host sessions bound to those rows disconnect on next request and re-authorize via the standard OAuth flow.
4. **Given** the upgrade has already run, **When** the operator loads another admin page, **Then** the upgrade callback is a no-op (no ALTER re-issued, no duplicate-index errors, no `SHOW WARNINGS` output) AND `acrossai_mcp_oauth_legacy_dcr_purged` does NOT fire a second time (nothing left to purge).
5. **Given** the upgrade has already run and the admin subsequently drops the `server_id` column manually via SQL and rewinds `db_version_key`, **When** `Main::reconcile_database_schemas()` fires on next admin request, **Then** the column is recreated and backfill is re-applied. Any new legacy-shape rows created between the drop and re-run are purged again.

### User Story 4 — User deletion still cascades site-wide (Priority: P1 — regression protection)

An admin deletes a WordPress user account via the standard WP Users screen. That user held OAuth tokens across multiple MCP servers (e.g., authorized Claude.ai on Server A and ChatGPT on Server B) AND was on the approved-users list for one or more connectors (F032 US5). Every one of that user's OAuth tokens, auth codes, AND connector-approval rows is revoked/deleted — regardless of which server issued them. The site-wide cleanup semantic per `FR-042` is preserved; F032's per-server scoping additions do NOT accidentally introduce a server filter on the user-deletion cascade path, and the FR-042 cascade is extended to include the new `wp_acrossai_mcp_connector_approved_users` table.

**Why this priority**: this behaviour was correct before F032 and must remain correct after. Regression here would leak an orphaned user's tokens after deletion, defeating the compliance guarantee of the WP user-lifecycle contract.

**Independent Test**: Seed a WordPress user with OAuth tokens on Server A AND Server B. Delete the user via `wp_delete_user()`. Assert BOTH servers' tokens for that user are `revoked = 1` and both servers' auth codes for that user are deleted.

**Acceptance Scenarios**:

1. **Given** a WordPress user holding non-revoked OAuth tokens on 2+ MCP servers, **When** `wp_delete_user()` is called, **Then** `UserLifecycle::on_user_deleted()` fires AND every token belonging to that user is revoked across all servers.
2. **Given** the same setup, **When** the user is deleted, **Then** every auth code belonging to that user is deleted across all servers.
3. **Given** F032's schema changes are in place, **When** the deletion cascade runs, **Then** `TokensQuery::revoke_by_user_id( int $user_id )` and `AuthCodesQuery::delete_by_user_id( int $user_id )` are called WITHOUT a `server_id` filter (site-wide cascade preserved).
4. **Given** the user had approval rows in `wp_acrossai_mcp_connector_approved_users` for multiple (server, connector) pairs, **When** `UserLifecycle::on_user_deleted()` fires, **Then** `ConnectorApprovedUsersQuery::delete_by_user_id( int $user_id )` is invoked WITHOUT any server_id or connector_slug filter — every approval row for that user is deleted site-wide.

### User Story 5 — Admin manages a first-class Approved Users panel per connector (Priority: P2 — operator ergonomics)

An admin has enabled `require_admin_approval` on the Claude connector for Server A. A non-admin user attempts to connect Claude → they land on a pending-approval page. The admin opens the AI Connectors tab for Server A → sees a new "Approved Users" level-3 nav tab (present iff `require_admin_approval = true`). The panel has two sections: **Pending approvals** (list of users awaiting decision, with Approve + Deny buttons per row) and **Approved users** (widefat striped table of previously-approved users, with "Revoke approval" action per row + columns for approved-by admin + approval timestamp). The approved-user list is stored in a first-class BerlinDB table (`wp_acrossai_mcp_connector_approved_users`) — NOT the pre-F032 serialized wp_options blob — with a UNIQUE(server_id, connector_slug, user_id) presence constraint that makes duplicate approvals impossible at the SQL layer.

**Why this priority**: the pre-F032 wp_options storage had no relational integrity (dangling approvals after user deletion), no first-class query surface (LIST tables couldn't LEFT JOIN with wp_users), and the pending-approvals list was buried inside the Settings panel making it hard to discover. Promoting to a real table + a dedicated panel is a UX + data-integrity improvement.

**Independent Test**: Enable `require_admin_approval` on the Claude connector; verify the "Approved Users" panel appears in the level-3 nav. Non-admin user connects → verify the user appears in the panel's Pending section. Click Approve → verify a row is inserted into `wp_acrossai_mcp_connector_approved_users` AND the user disappears from Pending, appears in Approved. Click Revoke approval → verify the row is deleted from the table.

**Acceptance Scenarios**:

1. **Given** `require_admin_approval = true` on (Server A, Claude), **When** an admin loads the AI Connectors tab for Server A, **Then** a new level-3 nav tab labeled "Approved Users" is visible between "Connections" and "Settings".
2. **Given** `require_admin_approval = false` on the same connector, **When** the admin loads the same tab, **Then** the "Approved Users" nav tab is NOT rendered AND a direct URL to `?panel=approved-users` falls through to the default `?panel=generate` panel.
3. **Given** a fresh non-admin user attempts to connect the Claude connector, **When** they land on the pending-approval consent screen, **Then** their user_id is added to `pending_approvals` wp_option AND they appear in the "Approved Users" panel's Pending section.
4. **Given** a user is in the Pending section, **When** the admin clicks "Approve", **Then** a row is INSERTed into `wp_acrossai_mcp_connector_approved_users` for that (server, connector, user) triple AND the user disappears from Pending AND appears in the Approved section AND the `approved_by` column records the admin's user_id AND `approved_at` records the current UTC timestamp.
5. **Given** a user is in the Pending section, **When** the admin clicks "Deny", **Then** the user is removed from `pending_approvals` wp_option WITHOUT any row inserted into the Approved table AND the user must re-attempt the connect flow to try again.
6. **Given** a user is in the Approved section, **When** the admin clicks "Revoke approval", **Then** the row is DELETEd from `wp_acrossai_mcp_connector_approved_users` AND the `acrossai_mcp_connector_user_approval_revoked` action fires (see US6 for the cascade behaviour).

### User Story 6 — Revoke-approval cascades to active tokens (via filter opt-out) (Priority: P2 — operator ergonomics + security)

When an admin clicks "Revoke approval" for a user on the Approved Users panel (US5), the user's approval row is deleted AND, by default, every one of that user's active OAuth tokens for the same (server, connector) pair is also revoked. This closes the confusing UX gap where a revoked approval left a live session running until token expiry. Third-party plugins can `remove_action()` the default cascade listener OR set the `acrossai_mcp_connector_revoke_tokens_on_approval_revoked` filter to false to opt out — in which case the approval row is still deleted but the user's existing tokens stay active until they expire naturally.

**Why this priority**: revoking someone's approval and leaving their session live is a security foot-gun. The filter-based opt-out honors §V Extensibility Without Core Modification — projects that want the pre-F032 decoupled behaviour can restore it without a code fork.

**Independent Test**: Approve a user for Claude on Server A. Have them connect → verify at least one non-revoked row in `wp_acrossai_mcp_oauth_tokens` for that (server, client_id, user_id). As admin, click "Revoke approval" → verify (a) the approval row is deleted AND (b) the token rows are marked `revoked = 1`. Then: set `add_filter( 'acrossai_mcp_connector_revoke_tokens_on_approval_revoked', '__return_false' )`, re-approve, re-connect, revoke again → verify approval row deleted BUT tokens remain `revoked = 0`.

**Acceptance Scenarios**:

1. **Given** a user is on the Approved list for (Server A, Claude) AND has ≥1 non-revoked OAuth token bound to that (server, connector) pair, **When** the admin clicks "Revoke approval", **Then** the approval row is deleted AND the `acrossai_mcp_connector_user_approval_revoked` action fires with 4 args (`$server_id, $connector_slug, $user_id, $revoked_by`) AND the default listener revokes every matching non-revoked token (marks `revoked = 1`) AND fires `acrossai_mcp_manager_oauth_token_revoked` once per token with reason `approval_revoked`.
2. **Given** a listener registered `add_filter( 'acrossai_mcp_connector_revoke_tokens_on_approval_revoked', '__return_false' )`, **When** the admin clicks "Revoke approval", **Then** the approval row is deleted BUT no tokens are revoked (the cascade silently short-circuits).
3. **Given** a third-party plugin called `remove_action( 'acrossai_mcp_connector_user_approval_revoked', [ ConnectorAdminController::class, 'cascade_revoke_tokens_on_approval_revoked' ], 10 )`, **When** the admin clicks "Revoke approval", **Then** the approval row is deleted AND the cascade does not fire (equivalent to opt-out via filter).
4. **Given** the connector has both admin-generated clients (`server-{id}-` prefix) AND DCR-registered clients (Claude Desktop et al), **When** the cascade fires, **Then** BOTH categories of clients are enumerated and every matching token belonging to the user is revoked — matches the enumeration shape used by `mass_revoke_connector_tokens`.

### User Story 7 — "Revoke from all servers" nuclear button + annotated token counts (Priority: P3 — operator power)

An admin encounters a compromised AI-host session (e.g., Claude.ai leaked a token to a phishing site). They open any server's Connections panel → each client row now has a "Revoke from all servers" nuclear button (in addition to the per-server Revoke tokens + Delete client buttons from F032 US1). Clicking it revokes every non-revoked token for that `client_id` across every server on the site, without requiring the operator to visit each server tab individually. The button is a DELIBERATE carve-out from F032's per-server invariant — it MUST NOT fire the `acrossai_mcp_oauth_cross_server_attempted` action (that action is reserved for actual bypass ATTEMPTS, not this legitimate site-wide operator action).

Additionally, the Connections panel's "Active tokens" column now displays annotated counts (`2 (1 access · 1 refresh)`) instead of just a total, so the operator can distinguish between short-lived access tokens and long-lived refresh tokens at a glance.

**Why this priority**: Emergency response speed. If a client_id is compromised, the operator needs a one-click site-wide kill switch. Annotated token counts help the operator make informed decisions about whether the "Revoke tokens" button will kill just this-hour's access grant or a long-lived refresh grant.

**Independent Test**: Seed 2 servers with the same admin-generated `server-{id}-claude-ai-...` client_id + non-revoked tokens on both. Click "Revoke from all servers" on either row → verify BOTH servers' tokens for that client_id are `revoked = 1` AND the `acrossai_mcp_oauth_client_revoked_across_all_servers` action fires exactly once with 4 args AND the `acrossai_mcp_oauth_cross_server_attempted` action does NOT fire.

**Acceptance Scenarios**:

1. **Given** two servers each with a Claude.ai client + non-revoked tokens, **When** the admin clicks "Revoke from all servers" on either server's Claude row, **Then** BOTH servers' non-revoked tokens for that `client_id` are marked `revoked = 1` AND the response is 200 + `{ revoked_count, scope: 'all_servers' }`.
2. **Given** the same setup, **When** the admin clicks the new nuclear button, **Then** the `acrossai_mcp_oauth_client_revoked_across_all_servers` action fires exactly once with args `($client_id, $revoked_token_count, get_current_user_id(), time())` AND `acrossai_mcp_oauth_cross_server_attempted` does NOT fire (this is a legitimate site-wide op, not a bypass attempt).
3. **Given** the Connections panel is rendered, **When** the "Active tokens" column is displayed for a client with 1 access token + 1 refresh token, **Then** the displayed text is `"2 (1 access · 1 refresh)"` — verified via inspecting the rendered HTML.
4. **Given** the same button is clicked with a `client_id` that doesn't exist on any server (`revoked_count = 0`), **When** the request completes, **Then** the response is still 200 (idempotent no-op) but the aggregate action does not fire (nothing to observe).

### User Story 8 — Access Control blocks connect-time before credential issuance (Priority: P1 — security + UX)

A site uses the F015 Access Control v2 rule engine to restrict which WordPress user roles can access which MCP servers (e.g., subscribers may access Server A, editors may access Server A + B, only admins may access Server C). A subscriber attempts to connect Claude.ai to Server C. **Pre-F032 behaviour**: Claude.ai shows "connected" successfully, then every subsequent tool call silently returns 403 — invisible to the operator, confusing to the user. **F032 behaviour**: the OAuth `/authorize` flow gates on Access Control at consent time. If the user's roles don't satisfy the AC rules for the target server, the flow redirects with `error=access_denied` + a human-readable description BEFORE any auth code is issued. The user sees an immediate, clear denial. The CLI device-grant consent gate and Application Password generation gate use the same shared helper (`AcrossAI_MCP_Access_Control::user_has_server_access`), so the three connect-time surfaces produce consistent denial UX. Every denial fires `acrossai_mcp_access_control_denied` with context (`'oauth_authorize'`, `'cli_device_grant'`, or `'application_password'`) so operators can differentiate.

**Why this priority**: security (unauthorized users must not receive credentials, even short-lived ones) + UX (Claude showing "connected" then 403ing every tool call is a debugging nightmare). Deferring AC enforcement to tool-call time was a bug, not a design.

**Independent Test**: Configure F015 AC rules so subscribers may NOT access Server C. Log in as a subscriber, initiate Claude.ai OAuth flow targeting Server C → verify the flow redirects with `error=access_denied` BEFORE the consent screen renders, no auth code is issued, `acrossai_mcp_access_control_denied` fires with `context = 'oauth_authorize'`. Log in as an admin → verify the flow proceeds (vendor v2 hierarchy allows admins by default).

**Acceptance Scenarios**:

1. **Given** F015 AC rules deny subscribers on Server C, **When** a subscriber initiates an OAuth authorize request for Server C, **Then** the flow redirects to the AI host with `error=access_denied` + description "Your account does not have permission to connect to this MCP server. Contact a site administrator to request access." BEFORE the consent screen renders AND no row is inserted into `wp_acrossai_mcp_oauth_auth_codes` AND `acrossai_mcp_access_control_denied` fires with args `($user_id, $server_slug, null, 'oauth_authorize')`.
2. **Given** the same AC rules, **When** an admin initiates the same OAuth authorize request, **Then** the flow proceeds normally (vendor v2 hierarchy allows admins) AND the consent screen renders.
3. **Given** the AC package is not loaded (Composer package absent), **When** any user initiates OAuth authorize, **Then** `user_has_server_access()` returns `true` (fail-open per D19) AND the flow proceeds — the plugin remains usable without the optional AC package.
4. **Given** the OAuth authorize path resolves a server_id that doesn't correspond to a real `wp_acrossai_mcp_servers` row (Q2 race), **When** the AC helper is invoked, **Then** it returns `true` (fail-open) — same behaviour as the F015 tool-call gate.

### Edge Cases

- **Malformed `server-{id}-` prefix**: a legacy admin client row whose `client_id` has a corrupted prefix (e.g., `server-abc-claude-ai-xyz` — non-numeric server_id). The backfill `SUBSTRING_INDEX(...) AS UNSIGNED` cast produces 0, which does NOT match any real server_id. Row stays effectively legacy. Documented; no fatal.
- **Race condition — DCR request during upgrade**: a DCR request arrives BETWEEN plugin file replacement and `Main::reconcile_database_schemas()` firing on `admin_init@3`. Per FR-028, the DCR endpoint MUST check for `server_id` column presence via `INFORMATION_SCHEMA` before INSERT; if the column is absent (pre-migration schema) the endpoint returns 503 `service_unavailable`, and the AI-host client retries after the operator loads any admin page (which triggers the migration). Once the column exists, the DCR handler proceeds normally with `server_id` populated. No legitimate registration is silently destroyed by the auto-purge step (per plan-review SEC-032-005 remediation).
- **Mid-migration crash**: PHP fatal error between "ADD COLUMN" and "add composite UNIQUE". Next admin request re-enters `upgrade_to_<v>()` — the column exists (step 1 idempotent) and the composite is missing (step 2 not yet done) → step 2 completes. `db_version_key` is only stamped on full success, so the upgrade retries until every step completes.
- **RFC 8707 `resource` URL uses a non-canonical form** (trailing slash, different port, IPv6 host): `resolve_server_id_from_resource_url()` normalises via the same route-matching helper `CurrentServerHolder::capture_from_request()` uses. Consistent behaviour across DCR + request-time server context.
- **DCR client already exists globally (pre-F032 orphan) and new DCR request arrives**: cannot happen after the upgrade has run — all legacy `server_id = NULL` rows are purged in the same upgrade callback that adds the column. The new DCR request creates a fresh server-scoped row without any legacy-orphan conflict. If the request arrives during the narrow window BEFORE the upgrade fires (fresh admin session on a stale install), it falls into the pre-migration schema path (row created without `server_id`) and is itself purged when the upgrade runs on the next admin request.
- **REST request body includes both `server_id` and a `client_id` that starts with `server-{other_id}-`**: the server_id in the request body is authoritative — the endpoint looks up the client by `find_by_client_id_and_server_id( client_id, server_id )`. If no match → 403. The prefix in the `client_id` string is NEVER re-parsed after F032; only the DB `server_id` column matters.

### Functional Requirements

- **FR-001**: The `wp_acrossai_mcp_oauth_clients` table MUST gain a `server_id BIGINT UNSIGNED NOT NULL` column via BerlinDB's D28 3-part contract (bump `$version`; register `$upgrades['<new>' => 'upgrade_to_<new>']`; idempotent `upgrade_to_<new>()` method). The column is added as `NULL` transiently during the upgrade callback so backfill + purge can run, then MODIFYed to `NOT NULL` as the final step per FR-026. Schema.php declares the final `NOT NULL` state.
- **FR-002**: The `wp_acrossai_mcp_oauth_tokens` table MUST gain the same `server_id BIGINT UNSIGNED NOT NULL` column via the same D28 contract, plus a composite `KEY(server_id, client_id)` index for query performance.
- **FR-003**: The `wp_acrossai_mcp_oauth_auth_codes` table MUST gain the same `server_id BIGINT UNSIGNED NOT NULL` column via the same D28 contract.
- **FR-004**: The `oauth_clients` table's uniqueness constraint MUST change from standalone `UNIQUE(client_id)` to composite `UNIQUE(client_id, server_id)`. The upgrade MUST add the composite index BEFORE dropping the standalone one so the table is never unconstrained mid-migration.
- **FR-005**: On upgrade, admin-generated clients (rows whose `client_id` matches `^server-(\d+)-`) MUST have `server_id` backfilled from the prefix via idempotent `UPDATE ... WHERE server_id IS NULL AND client_id LIKE 'server-%'`. The UPDATE MUST additionally guard against phantom-server assignments — the parsed server_id MUST be verified to exist in `wp_acrossai_mcp_servers` via `AND CAST(...) IN (SELECT id FROM wp_acrossai_mcp_servers)` (per plan-review SEC-032-003 remediation, 2026-07-21). Rows whose parsed server_id doesn't match a real server are left with `server_id IS NULL` and correctly PURGED in the FR-007 step alongside legacy DCR rows.
- **FR-006**: On upgrade, `oauth_tokens` and `oauth_auth_codes` MUST have `server_id` backfilled via `JOIN oauth_clients ON client_id` — copying `oauth_clients.server_id` to the token/auth_code row where the client row has a resolved server_id.
- **FR-007**: After the backfill step, the `oauth_clients` upgrade callback MUST purge every row where `server_id IS NULL` (legacy pre-F032 DCR clients whose `client_id` does NOT match the `server-{id}-` prefix). Corresponding rows in `oauth_tokens` and `oauth_auth_codes` are purged by their own upgrade callbacks after their respective JOIN backfill step (see FR-024). The purge is unconditional — rows with live non-revoked tokens are NOT preserved (accepted tradeoff per 2026-07-21 clarification Q3). Live AI-host sessions bound to purged rows disconnect on next request and re-authorize via the standard OAuth flow.
- **FR-008**: Every OAuth `Query` class method that reads or mutates client/token/auth_code rows MUST accept a `server_id` parameter to filter results — except `TokensQuery::revoke_by_user_id( int $user_id )` and `AuthCodesQuery::delete_by_user_id( int $user_id )` which MUST remain server-neutral (user-deletion cascade is site-wide per FR-042 from Feature 021).
- **FR-009**: `TokensQuery::get_active_user_ids_by_client_id( string $client_id )` MUST be renamed to `get_active_user_ids_by_client_id_and_server_id( string $client_id, int $server_id )` and gain a required `server_id` param — closes the read-side cross-server display leak in the AI Connectors tab's "authorized users" listing.

- **FR-011**: `TokenController::handle_authorization_code()` MUST copy `server_id` from the exchanged auth-code row onto the emitted token row.
- **FR-012**: `TokenController::handle_refresh_token()` MUST copy `server_id` from the prior token row onto the emitted refreshed token row.
- **FR-013**: `TokenController` MUST verify at code-exchange time that the client row's `server_id` matches the auth-code row's `server_id`; on mismatch, return `invalid_grant` (defense-in-depth against data corruption).
- **FR-014**: `ClientRegistrationController::handle_register()` (the DCR endpoint) MUST resolve `server_id` from the RFC 8707 `resource` param in the DCR request body and persist it on the new client row. If the resource URL does not resolve to a known MCP server, return `WP_Error( 'invalid_target', 400 )` and do NOT create the row.
- **FR-015**: `ClientRegistrationController::handle_admin_generate()` MUST persist `server_id` on the new admin-generated client row (already known from the admin form context).
- **FR-016**: `ConnectorAdminController::handle_revoke_client_tokens()` MUST require a `server_id` parameter in the request. On receiving a `403 acrossai_mcp_oauth_cross_server` response, it MUST surface a distinct error message ("This action can only be performed for the server that owns this client — refresh the page and try again.") — distinct from generic 403 so operators understand the cause.

- **FR-025**: The README.txt Unreleased changelog entry AND the plugin's user-facing update-notice screen MUST include a prominent warning that upgrading to the F032-shipping version will disconnect any live AI-host session bound to a pre-F032 DCR client, and that affected users will need to re-run the OAuth authorize flow to reconnect.
- **FR-026**: After the backfill (FR-005/FR-006), purge (FR-007), and composite-UNIQUE swap (FR-004) steps have completed within each table's upgrade callback, the callback MUST execute `ALTER TABLE ... MODIFY server_id BIGINT UNSIGNED NOT NULL` as the final step. The MODIFY is guarded by an `INFORMATION_SCHEMA.COLUMNS.IS_NULLABLE = 'YES'` check for idempotency (re-runs are no-ops). Schema.php MUST declare `server_id` as `NOT NULL` on all three tables so BerlinDB's SHOW CREATE TABLE match check stays green on subsequent boots. - **FR-027**: `ClientRegistrationController::resolve_server_id_from_resource_url( string $resource ): int` MUST perform a two-step check: (a) origin verification — reject any `resource` URL whose origin (scheme + host + port) does not match `home_url()` (or `network_home_url()` on multisite installs, though F032 is single-site only per `multisite_support: false`). On origin mismatch return 0 (which causes the DCR handler to emit `WP_Error( 'invalid_target', 400 )` with distinct log context `resource_url_origin_mismatch`). (b) Path resolution against `MCPServerQuery` via the existing route-matching helper from `CurrentServerHolder::capture_from_request()` for consistent normalization (trailing slash, port differences, IPv6 hosts). Prevents URL-confusion / SSRF-adjacent attacks where an attacker sends a DCR request whose `resource` URL points at an evil origin but whose path structurally matches our wp-json route pattern (per plan-review SEC-032-002 remediation, 2026-07-21).
- **FR-028**: `ClientRegistrationController::handle_register()` MUST verify the `server_id` column exists on `wp_acrossai_mcp_oauth_clients` via `INFORMATION_SCHEMA.COLUMNS` (per-request cached) BEFORE the INSERT. Prevents legitimate DCR registrations from being silently destroyed by the F032 auto-purge step (per plan-review SEC-032-005 remediation, 2026-07-21). Compliant AI-host clients that honor `Retry-After` will succeed on retry once migration completes.

### Database / Storage

**Custom DB tables (existing) gain one column each**:
- `{$wpdb->prefix}acrossai_mcp_oauth_clients` — new column `server_id BIGINT UNSIGNED NOT NULL` (final state) + composite `UNIQUE(client_id, server_id)` replacing `UNIQUE(client_id)`.

- `{$wpdb->prefix}acrossai_mcp_oauth_auth_codes` — new column `server_id BIGINT UNSIGNED NOT NULL` (final state).

**New custom DB table (per FR-029)**:
- `{$wpdb->prefix}acrossai_mcp_connector_approved_users` — 6 columns (`id`, `server_id`, `connector_slug`, `user_id`, `approved_by`, `approved_at`). 4 indexes: `PRIMARY(id)`, `UNIQUE(server_id, connector_slug, user_id)` (presence constraint), `KEY(server_id, connector_slug)` (panel-listing lookup), `KEY(user_id)` (UserLifecycle cascade lookup). BerlinDB Kern Table + Schema + Row + Query classes under `includes/Database/ConnectorApprovedUsers/`. Table subclass uses leading-`\` FQN per DEC-BERLINDB-SUBCLASS-NO-USE-COLLISION, overrides `maybe_upgrade()` with F011 phantom-version guard. `$version = '1.0.0'`. `$db_version_key = 'acrossai_mcp_connector_approved_users_db_version'`. Fresh-install-only per D21 (no migration from pre-F032 wp_options).

**Upgrade callback ordering per table** (each idempotent, gated by INFORMATION_SCHEMA checks):
1. ADD COLUMN `server_id BIGINT UNSIGNED NULL` if column missing.
2. Backfill (prefix parse for clients, JOIN for tokens/auth codes).
3. Purge remaining `server_id IS NULL` rows (legacy pre-F032 DCR + descendants).
4. (OAuthClients only) ADD composite `UNIQUE(client_id, server_id)` if missing, then DROP standalone `UNIQUE(client_id)` if present.
5. MODIFY `server_id` to `NOT NULL` if `INFORMATION_SCHEMA.COLUMNS.IS_NULLABLE = 'YES'`.
6. (OAuthClients only, last) Fire `do_action( 'acrossai_mcp_oauth_legacy_dcr_purged', $clients_deleted, $tokens_deleted, $auth_codes_deleted )` if any purge counted > 0.

Steps within a callback run in a single request; `db_version_key` is only stamped on full-callback success, so a mid-migration crash retries every step on the next admin request.

**Upgrade mechanism**: BerlinDB `$upgrades` callbacks on each Table subclass, fired by `Main::reconcile_database_schemas()` on `admin_init@3` (existing wiring from F029). Every callback is idempotent per D28 contract.

**Data preservation**: table names + `db_version_key` option names preserved byte-for-byte. Backwards-compatible: existing rows continue to work; admin clients backfill from prefix; tokens/auth codes backfill via JOIN; DCR legacy rows remain with `server_id = NULL`.

### Key Entities

## Assumptions

- BerlinDB Core 3.0's `$upgrades` mechanism + `Main::reconcile_database_schemas()` on `admin_init@3` (wired in F029) are the canonical column-add path. F032 is the third feature to use them (after F029 `CliAuthLog` + `MCPServer` and F030 `MCPServer`); the pattern is well-tested.
- Admin-generated OAuth clients have followed the `server-{id}-{slug}-{rand}` naming convention since F021. Any client_id NOT matching this shape is either DCR-registered (Claude.ai, ChatGPT, Cursor, Cline) OR pre-F021 legacy (unlikely on any live install).
- The RFC 8707 `resource` parameter is already parsed at authorize time (F021's audience-binding) — F032 reuses that parsing, doesn't introduce new URL-normalisation logic.
- Legacy DCR rows with `server_id = NULL` may represent live OAuth sessions from real AI hosts. F032 deliberately purges them during the upgrade callback (accepted tradeoff per 2026-07-21 clarification Q3, A-aggressive form) — this silently disconnects any live AI-host session bound to those rows on the next request. Affected users re-authorize via the standard OAuth flow. README + release-note MUST prominently warn operators before upgrade per FR-025. The tradeoff is intentional: the alternative (preserving legacy rows as invisible orphans, or shipping a follow-up WP-CLI purge) leaves the tables in a mixed post-migration state indefinitely, complicating every future query and re-opening the read-side leak surface F032 exists to close.
- The `wp_acrossai_mcp_oauth_audit` table already records `server_id` per F029 — F032 does NOT touch it.
- Multisite is out of scope for this increment (matches plugin-wide `multisite_support: false`).
- Any active site currently exploiting the cross-server bypass semantically is broken behavior, not a documented feature. F032's 403 response to previously-successful cross-server calls is a fix, not a regression. Pre-flight security-review attestation (SEC-032-001 / T001 per the planning doc) captures this at plan phase as awareness only; it is NOT a fallback trigger for shipping behind a feature flag. F032 ships unconditionally with no `acrossai_mcp_manager_oauth_per_server_scoping_enabled` toggle — rollback is via composer package downgrade if operationally required.
